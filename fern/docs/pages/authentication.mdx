All ListBee API requests authenticate with an API key passed as a Bearer token. Keys start with `lb_` and are tied to one account.

```
Authorization: Bearer lb_...
```

Missing or invalid keys return `401`.

## Two ways to get a key

1. **Bootstrap API** (agent-driven) — the agent creates an account, a human pastes a 6-digit OTP, the agent gets a key. No console visit required.
2. **Console** (human-driven) — sign in at [console.listbee.so](https://console.listbee.so), create keys in the API Keys page.

Both paths produce the same account. If an account already exists for that email, bootstrap reuses it.

## Bootstrap flow

Two API calls plus a poll. The agent drives; the human hands over one 6-digit code and completes Stripe Connect in a browser.

### Step 1: Send OTP

<CodeBlocks>
<CodeBlock title="Python">
```python
from listbee import ListBee

client = ListBee()  # no api_key needed for bootstrap
start = client.bootstrap.start(email="seller@example.com")
# Tell the human: check your email for a 6-digit code.
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/bootstrap/start \
  -H "Content-Type: application/json" \
  -d '{"email": "seller@example.com"}'
```
</CodeBlock>
</CodeBlocks>

Response:

```json
{
  "object": "bootstrap_session",
  "bootstrap_token": "bst_abc123",
  "account_id": "acc_01J3K4M5N6P7Q8R9S0T1U2V3W4",
  "otp_expires_at": "2026-04-17T12:10:00Z"
}
```

The `bootstrap_token` is single-use and expires in 10 minutes. The `account_id` is stable — you can start polling it immediately.

### Step 2: Verify and collect the key

The human reads the 6-digit code from their email and pastes it back. The agent submits it with the `bootstrap_token`.

<CodeBlocks>
<CodeBlock title="Python">
```python
result = client.bootstrap.verify(
    bootstrap_token=start.bootstrap_token,
    otp_code="123456",
)
api_key = result.api_key                        # lb_... — save immediately, not shown again
onboarding_url = result.stripe_onboarding_url   # hand to the human
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/bootstrap/verify \
  -H "Content-Type: application/json" \
  -d '{"bootstrap_token": "bst_abc123", "otp_code": "123456"}'
```
</CodeBlock>
</CodeBlocks>

Response:

```json
{
  "object": "bootstrap_result",
  "account_id": "acc_01J3K4M5N6P7Q8R9S0T1U2V3W4",
  "api_key": "lb_01J3K4M5N6P7Q8R9S0T1U2V3W4X5Y6Z7",
  "stripe_onboarding_url": "https://connect.stripe.com/setup/s/...",
  "readiness": {
    "operational": false,
    "actions": [
      {
        "code": "stripe_connect_required",
        "kind": "human",
        "priority": "required",
        "message": "Complete Stripe Connect onboarding to accept payments.",
        "resolve": {"method": "redirect", "url": "https://connect.stripe.com/setup/s/..."}
      }
    ],
    "next": "stripe_connect_required"
  }
}
```

The API key is returned once. Listings can already be created and published — payment collection unlocks once Stripe Connect completes.

### Step 3: Poll until Stripe is connected

Hand `stripe_onboarding_url` to the human. While they complete onboarding in a browser, the agent polls:

<CodeBlocks>
<CodeBlock title="Python">
```python
import time

while True:
    status = client.bootstrap.poll(result.account_id)
    if status.ready:
        break
    time.sleep(5)
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl https://api.listbee.so/v1/bootstrap/acc_01J3K4M5N6P7Q8R9S0T1U2V3W4 \
  -H "Authorization: Bearer lb_..."
```
</CodeBlock>
</CodeBlocks>

`ready: true` means `readiness.operational` is `true` — the account can accept payments.

### One-liner helper

The Python and TypeScript SDKs bundle the full flow in `bootstrap.run()`:

```python
api_key = client.bootstrap.run(
    "seller@example.com",
    on_otp=lambda: input("Enter OTP: "),
    on_human_action=lambda url: print(f"Visit: {url}"),
)
```

`run()` calls `start`, prompts for the OTP via `on_otp`, calls `verify`, hands the Stripe URL to `on_human_action`, and polls until the account is ready. It returns the API key.

## Revoking a key

An agent can destroy its own key without a human in the loop — useful if the key is leaked or the host is compromised.

<CodeBlocks>
<CodeBlock title="Python">
```python
client.api_keys.self_revoke()
# The calling key is dead the moment this returns.
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/api-keys/self-revoke \
  -H "Authorization: Bearer lb_..."
```
</CodeBlock>
</CodeBlocks>

`POST /v1/api-keys/self-revoke` revokes the key used to authenticate the call. It cannot list, read, or touch any other key on the account. Multi-key management (list, create, revoke a different key) is console-only at [console.listbee.so](https://console.listbee.so).

## Threat model for the OTP-over-chat handoff

The bootstrap flow deliberately funnels the OTP code through the human's chat with the agent: the agent reads it, and that's required — it submits it to `/verify`. The code is short-TTL (10 min), single-use, and scoped to ListBee via the paired `bootstrap_token`. Agents should not persist or log the OTP after verify succeeds.

If this model doesn't fit, the human can bootstrap through the console and hand the agent a key manually.
