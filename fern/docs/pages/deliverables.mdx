A deliverable is the thing being sold. One deliverable per order — either a URL the buyer is redirected to, or a block of Markdown text rendered on the unlock page.

ListBee does not host files. You host the content at a URL you control, or you deliver text that ListBee renders.

## Types

| Type   | Input                       | What the buyer sees                    | Who hosts |
|--------|-----------------------------|----------------------------------------|-----------|
| `url`  | any `https://` URL          | Browser redirects to your URL          | You       |
| `text` | Markdown string             | Rendered Markdown on the unlock page   | ListBee renders it |

**URL** — ListBee redirects the buyer to your URL on unlock. Use for anything with a link: Notion pages, signed download URLs, Google Drive share links, community invites, Zoom rooms. ListBee does not validate, proxy, or cache the URL — if it goes offline, the buyer sees a broken page. Access control on the destination is your responsibility.

**Text** — ListBee renders the string as Markdown on the unlock page. Use for license keys, access codes, short instructions, AI-generated reports, or any content that fits on one page. Supports headings, lists, bold, italic, links, code blocks.

## Builders

The Python and TypeScript SDKs expose `Deliverable.url()` and `Deliverable.text()` helpers so you don't hand-build the shape.

<CodeBlocks>
<CodeBlock title="Python">
```python
from listbee import Deliverable

Deliverable.url("https://example.com/secret-download")
Deliverable.text("Your license key: ABCD-1234-EFGH-5678")
```
</CodeBlock>
<CodeBlock title="curl">
```json
{"type": "url",  "value": "https://example.com/secret-download"}
{"type": "text", "value": "Your license key: ABCD-1234-EFGH-5678"}
```
</CodeBlock>
</CodeBlocks>

## Where deliverables go

There are two placement points, driven by the listing's `fulfillment_mode`:

### On the listing (`static` mode)

Bake the deliverable into the listing. ListBee delivers it automatically at `order.paid`.

<CodeBlocks>
<CodeBlock title="Python">
```python
client.listings.create(
    name="SEO Playbook 2026",
    price=1900,
    fulfillment_mode="static",
    deliverable=Deliverable.url("https://example.com/seo-playbook.pdf"),
)
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/listings \
  -H "Authorization: Bearer lb_..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "SEO Playbook 2026",
    "price": 1900,
    "fulfillment_mode": "static",
    "deliverable": {"type": "url", "value": "https://example.com/seo-playbook.pdf"}
  }'
```
</CodeBlock>
</CodeBlocks>

Updating the deliverable on a `static` listing only affects orders placed after the update — existing orders snapshot their deliverable at fulfillment time.

### On the order (`async` mode)

Leave the listing deliverable empty. After each order, the agent generates content and pushes it via `POST /v1/orders/{id}/fulfill`.

<CodeBlocks>
<CodeBlock title="Python">
```python
client.orders.fulfill(
    order_id="ord_9xM4kP7nR2qT5wY1",
    deliverable=Deliverable.text("Your personalized report: ..."),
)
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1/fulfill \
  -H "Authorization: Bearer lb_..." \
  -H "Content-Type: application/json" \
  -d '{"deliverable": {"type": "text", "value": "Your personalized report: ..."}}'
```
</CodeBlock>
</CodeBlocks>

`fulfill` is idempotent on the same deliverable — calling it twice with the same payload is a no-op.

## How buyers receive it

On payment, ListBee generates an `unlock_url` on the order (a permanent bearer link) and emails it to the buyer. The buyer can open the link any time:

- `url` deliverables → HTTP 302 redirect to the stored URL
- `text` deliverables → rendered Markdown page

If the order is `paid` but not yet `fulfilled` (async mode, agent hasn't pushed content), the unlock page shows a "processing" state and polls until the content is delivered.

`unlock_url` has no expiry, no view counter, and no per-buyer revocation. Treat it like a bearer token — a buyer who shares the link shares the content. If you need stricter access control, host the content at a URL that enforces it and deliver that URL.

## Limits

| Limit                  | Value                       |
|------------------------|-----------------------------|
| Deliverables per order | Exactly one                 |
| URL length             | Must be a valid `https://`  |
| Text length            | 10,000 characters           |
