ListBee is organized around a small set of entities. Understanding their relationships makes API responses easier to navigate and agent workflows easier to design.

## Hierarchy

```
Account

‚îÇ   ‚îî‚îÄ‚îÄ Deliverable (one, optional, for static mode)
‚îú‚îÄ‚îÄ Orders
‚îÇ   ‚îî‚îÄ‚îÄ Deliverable (one, set at fulfillment time)
‚îú‚îÄ‚îÄ Events
‚îî‚îÄ‚îÄ ApiKeys
```

An **Account** is the root entity. Everything belongs to an account. One account maps to one Stripe Connect onboarding, one billing subscription, and one brand (display name, bio, avatar).

**Identities** authenticate the human behind an account ‚Äî currently email OTP; Google and GitHub OAuth are stubbed for later. An account can have multiple identities that resolve to the same owner.

**Listings** are the sellable items an account creates. Each listing has a hosted product page at `buy.listbee.so/{short_code}`, a per-listing `signing_secret` for webhook verification, and an optional `agent_callback_url`. A listing in `static` mode carries a pre-baked deliverable; `async` mode leaves delivery to `POST /v1/orders/{id}/fulfill`.

**Orders** are created when Stripe confirms payment. They progress from `paid` to `fulfilled`. Each order holds an immutable snapshot of the listing and seller at purchase time, plus a permanent `unlock_url` the buyer uses to retrieve the deliverable.

**Events** are persisted records of domain events (`order.paid`, `order.fulfilled`, `order.refunded`, `order.disputed`). Queryable via `GET /v1/events` for reconciliation, and the same shape is POSTed to `agent_callback_url` for real-time delivery.

**ApiKeys** are the machine credentials used to authenticate API requests. The `lb_` value is returned once at creation ‚Äî stored as a hash. Entity ID uses the `lbk_` prefix.

## ID prefix table

| Prefix | Entity | Example |
|--------|--------|---------|
| `acc_` | Account | `acc_7kQ2xY9mN3pR5tW1vB8a` |
| `idn_` | Identity | `idn_7kQ2xY9mN3pR5tW1vB8a` |
| `lst_` | Listing | `lst_7kQ2xY9mN3pR5tW1vB8a` |
| `ord_` | Order | `ord_7kQ2xY9mN3pR5tW1vB8a` |
| `evt_` | Event | `evt_7kQ2xY9mN3pR5tW1vB8a` |
| `lbk_` | ApiKey (entity) | `lbk_7kQ2xY9mN3pR5tW1vB8a` |
| `lb_` | API key bearer token | `lb_abcdefghijklmnopqrstuvwx` |
| `bst_` | Bootstrap token | `bst_abc123` |

IDs use base62 encoding over a UUID4 and are always 22 characters after the prefix.

## Listing states

| Status | Description |
|--------|-------------|
| `draft` | Created but not published. Not accessible to buyers. |
| `published` | Live. Product page accepts orders. |
| `archived` | Terminal. Product page returns 404. No re-publish. |

Buyability is derived from status plus account readiness. A `published` listing on an account without Stripe Connect shows a disabled buy button until onboarding completes.

## Order states

| Status | Description |
|--------|-------------|
| `paid` | Stripe confirmed payment; order exists |
| `fulfilled` | Deliverable attached and emailed to buyer |

Two orthogonal axes run in parallel:

- `payment_status`: `paid` or `refunded`
- `dispute_status`: `null`, `needs_response`, `under_review`, `won`, `lost`

## Deliverable types

| Type   | Delivered as |
|--------|--------------|
| `url`  | 302 redirect from the unlock page to the stored URL |
| `text` | Rendered Markdown on the unlock page |

## Event types

| Event             | When |
|-------------------|------|
| `order.paid`      | Payment confirmed by Stripe |
| `order.fulfilled` | Deliverable attached (static at paid, async via `/fulfill`) |
| `order.refunded`  | Full refund issued |
| `order.disputed`  | Buyer opened a Stripe dispute |

Both `order.paid` and `order.fulfilled` fire for every order. Agents can rely on `order.fulfilled` as the universal delivery signal.
