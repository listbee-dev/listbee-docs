ListBee is organized around a small set of entities. Understanding their relationships makes API responses easier to navigate and agent workflows easier to design.

## Hierarchy

An **Account** is the root entity. Everything belongs to an account. One account maps to one Stripe Connect onboarding, one billing subscription, and one brand (display name, bio, avatar).

**Listings** are the sellable items an account creates. Each listing has a hosted product page at `buy.listbee.so/{slug}`. Listings belong to an account and can have up to three deliverables.

**Orders** are created when a buyer submits a checkout form. They progress through a state machine from `pending` (form submitted, awaiting payment) to a terminal state (`fulfilled`, `canceled`, or `failed`). Each order holds a snapshot of the listing and seller at time of purchase.

**AccessGrants** are created automatically when an order reaches `fulfilled`. They contain a signed download token valid for 48 hours and up to 10 downloads. Buyers access their content through this token ‚Äî ListBee never emails download links directly.

**Webhooks** are outbound HTTP endpoints you configure to receive events. Each webhook filters by event type and has a signing secret (`whsec_...`) for payload verification.

**WebhookEvents** are individual delivery attempts. If delivery fails, ListBee retries with exponential backoff. You can inspect event history and retry manually via the API.

**ApiKeys** are the machine credentials used to authenticate API requests. The `lb_` value is shown once at creation. The stored entity has prefix `ak_` and stores only a hashed form.

IDs use base62 encoding over a UUID4 and are always 22 characters after the prefix.

## Listing states

| Status | Description |
|--------|-------------|
| `draft` | Created but not published. Not accessible to buyers. |
| `published` | Live. Product page accepts orders. |
| `archived` | Taken offline. Product page returns 404. |

A listing can be published once the account has completed Stripe Connect onboarding. Listings with deliverables deliver automatically on payment. Listings without deliverables rely on a webhook or `fulfillment_url` callback for delivery ‚Äî both paths satisfy the readiness check.

## Order states

| Status | Description |
|--------|-------------|
| `pending` | Buyer submitted form, awaiting payment |
| `paid` | Stripe confirmed payment; delivery in progress |
| `fulfilled` | Buyer received content |
| `canceled` | Abandoned or manually canceled (24h TTL on `pending`) |
| `failed` | Payment or delivery error |

## Deliverable types

Each listing can have up to three deliverables in any combination:

| Type | What it is | Delivered as |
|------|-----------|-------------|
| `FILE` | Uploaded file (PDF, ZIP, etc.) | Signed download URL (48h, 10 downloads) |
| `URL` | External link | Redirect on the download page |
| `TEXT` | Raw text (license key, instructions, etc.) | Rendered on the download page |

## Webhook events

| Event | When |
|-------|------|
| `order.paid` | Buyer's payment succeeded |
| `order.fulfilled` | Order delivery confirmed |
| `order.refunded` | Refund issued |
| `order.canceled` | Order abandoned or canceled |
| `listing.published` | Listing went live |

Both `order.paid` and `order.fulfilled` fire for every order. Agents relying on `order.fulfilled` as the delivery signal will work consistently regardless of how the listing is configured.
