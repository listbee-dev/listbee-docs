Fulfillment is what happens after `checkout.session.completed` fires on Stripe. ListBee creates the order, captures snapshots, and either delivers immediately or waits for the agent to push content.

## Two modes

The listing's `fulfillment_mode` decides:

- **`static`** — the listing carries a pre-baked `deliverable`. ListBee delivers it at `order.paid`, fires `order.fulfilled`, transitions the order to `fulfilled`. No agent code runs.
- **`async`** — the listing has no deliverable. ListBee fires `order.paid`. The agent generates content and calls `POST /v1/orders/{id}/fulfill`. ListBee emails the unlock URL and fires `order.fulfilled`.

Pick static for fixed content (ebooks, links, license keys). Pick async for anything generated per-order (personalized reports, AI output, invite codes).

## Delivery channels

There are two ways to learn about orders:

1. **Per-listing webhook** — set `agent_callback_url` on the listing. ListBee POSTs signed events to that URL.
2. **Polling** — `GET /v1/orders` or `GET /v1/events`. Useful as a fallback or primary channel for stateless workers.

Both can be used together. The event log (`/v1/events`) is always populated regardless of webhook delivery outcome.

## Webhook events

Delivered to `agent_callback_url` (when set):

| Event              | Fires when                                         |
|--------------------|----------------------------------------------------|
| `order.paid`       | Stripe confirmed payment, order exists             |
| `order.fulfilled`  | Deliverable attached (static or via `/fulfill`)    |
| `order.refunded`   | Full refund issued                                 |
| `order.disputed`   | Buyer opened a Stripe dispute                      |

Static listings fire `order.paid` and `order.fulfilled` back-to-back. Async listings fire `order.paid`, then `order.fulfilled` whenever the agent calls `/fulfill`.

Listing and order `metadata` both ride through — your agent doesn't need a state lookup to respond.

## Verifying signatures

Each POST to `agent_callback_url` includes:

- `X-ListBee-Signature: t=1713436980,v1=abc123...` — timestamp + HMAC-SHA256 hex
- `X-ListBee-Event: order.paid` — event type for quick routing

The signing key is the listing's `signing_secret`. Signed payload is `f"{timestamp}.{raw_body}"`. Reject requests with `|now - t| > 300s` to defend against replay.

<CodeBlocks>
<CodeBlock title="Python">
```python
import hmac
import hashlib
import time

def verify(body: bytes, header: str, secret: str, tolerance: int = 300) -> bool:
    parts = dict(p.split("=", 1) for p in header.split(","))
    ts = int(parts["t"])
    if abs(time.time() - ts) > tolerance:
        return False
    expected = hmac.new(
        secret.encode(),
        f"{ts}.".encode() + body,
        hashlib.sha256,
    ).hexdigest()
    return hmac.compare_digest(expected, parts["v1"])

# In your handler:
raw_body = request.body()                     # raw bytes, not parsed JSON
header = request.headers["X-ListBee-Signature"]
if not verify(raw_body, header, LISTING_SIGNING_SECRET):
    return 401
```
</CodeBlock>
<CodeBlock title="curl">
```bash
# No curl verification flow — signatures are verified on the receiving server
# using the per-listing signing_secret returned at listing creation.
```
</CodeBlock>
</CodeBlocks>

The `signing_secret` is returned exactly once — on `POST /v1/listings` or a `PATCH` rotation. Subsequent reads only expose `signing_secret_preview` (last 4 chars). See [Listings](/listings#signing-secret).

You can also supply your own secret at creation time:

```python
client.listings.create(
    name="...",
    price=1900,
    signing_secret="my-pre-shared-secret",   # any string, typically 32+ chars
)
```

This enables stateless agents: bake a known secret into the worker, create the listing with that secret, verify webhooks without ever reading back ListBee state.

## Retry schedule

If `agent_callback_url` returns non-2xx or times out, ListBee retries with jitter:

```
attempt 1:  0s (immediate)
attempt 2:  30s  ± 20%
attempt 3:  5m   ± 20%
attempt 4:  30m  ± 20%
attempt 5:  2h   ± 20%
attempt 6:  6h   ± 20%
attempt 7:  24h max
```

Seven attempts spanning ~24 hours. Beyond this, the event remains queryable via [`GET /v1/events`](/events) for manual reconciliation.

## Circuit breaker

If a destination URL fails more than 50 times within 1 hour, ListBee opens a breaker for 15 minutes: new events still queue (nothing is dropped) but scheduling is skipped. The breaker closes automatically on the first successful delivery or when the cooldown expires.

This protects your server from cascading failures during an outage and stops ListBee from hammering a dead endpoint.

## Per-URL rate limiting

Deliveries to the same `agent_callback_url` are token-bucket rate-limited (10 req/s). Under a sudden burst of orders, deliveries queue with small delays instead of DDoS'ing the receiver.

## Redelivery

Forcibly requeue events for one order:

<CodeBlocks>
<CodeBlock title="Python">
```python
ack = client.orders.redeliver(order_id="ord_9xM4kP7nR2qT5wY1")
print(ack.scheduled_attempts)  # count of events re-queued
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1/redeliver \
  -H "Authorization: Bearer lb_..." \
  -H "Idempotency-Key: retry-2026-04-18-ord_9xM4kP7nR2qT5wY1"
```
</CodeBlock>
</CodeBlocks>

Sends `order.paid` (and `order.fulfilled` if applicable) with `attempt=1`, no initial backoff. Rate-limited per order (10/hour) and per API key (100/hour). If the listing has no `agent_callback_url`, `scheduled_attempts` is `0`.

## Stateless agent pattern

Agent callback + listing metadata + order metadata make a stateless worker tractable. Encode everything the agent needs at listing-create time, and the webhook carries it on every order:

```python
listing = client.listings.create(
    name="Custom Market Analysis",
    price=9900,
    fulfillment_mode="async",
    agent_callback_url="https://your-agent.example.com/generate",
    signing_secret="shared-secret-the-worker-already-knows",
    metadata={
        "report_type": "market_analysis",
        "data_sources": "bloomberg,sec_filings",
        "output_format": "pdf",
    },
)

# When order.paid fires, the webhook payload includes:
#   data.metadata                        → order-specific (from /fulfill, if set)
#   data.listing.metadata                → generation parameters
#   data.checkout_data                   → buyer-supplied fields
#   data.id                              → order to call /fulfill on
#
# The worker generates content → POST /v1/orders/{id}/fulfill → done.
# No database lookup, no state on the worker's side.
```
