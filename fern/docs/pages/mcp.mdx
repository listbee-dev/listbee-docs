ListBee ships an MCP server (`listbee-mcp`) that exposes the commerce API as tools any MCP-compatible agent can call. Agents can bootstrap an account, create listings, process orders, and fulfill async orders — without writing API integration code.

## Install

The server runs via `npx`. No global install required.

```bash
npx listbee-mcp
```

Requires Node.js 20 or later.

## Configuration

Set your API key as an environment variable before starting the server:

```bash
export LISTBEE_API_KEY=lb_...
npx listbee-mcp
```

### Claude Desktop

Add to `~/Library/Application Support/Claude/claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "listbee": {
      "command": "npx",
      "args": ["listbee-mcp"],
      "env": {
        "LISTBEE_API_KEY": "lb_..."
      }
    }
  }
}
```

### Cursor / Cline / other stdio clients

Same pattern — pass the command `npx listbee-mcp` and set `LISTBEE_API_KEY` in the process environment.

## Transports

The package ships two transports from the same codebase:

| Transport | How to use | Typical clients |
|-----------|-----------|-----------------|
| **stdio** | `npx listbee-mcp` | Claude Desktop, Cursor, Cline, Claude Code |
| **HTTP** | `mcp.listbee.so` | ChatGPT Apps, Claude API Connector, remote agents |

For the HTTP transport, point your client at `https://mcp.listbee.so`. Requests authenticate with `LISTBEE_API_KEY` in the `Authorization` header.

MCP discovery is available at [listbee.so/.well-known/mcp.json](https://listbee.so/.well-known/mcp.json).

## Available tools

Grouped by resource:

**Bootstrap**
- `bootstrap_start` — send OTP to a human's email, begin onboarding
- `bootstrap_verify` — verify OTP, receive API key and Stripe onboarding URL
- `bootstrap_poll` — poll account readiness until Stripe is connected

**Listings**
- `create_listing` — create a draft or published listing with a deliverable, metadata, signing secret
- `get_listing`, `list_listings`, `update_listing`, `delete_listing`
- `publish_listing` — draft → published

**Orders**
- `list_orders` — paginate orders, filter by status (`paid`, `fulfilled`) or listing
- `get_order`
- `fulfill_order` — attach a deliverable to an async order
- `refund_order` — full refund
- `order_redeliver` — requeue webhook events for a single order

**Account**
- `get_account`, `update_account`, `delete_account`
- `start_stripe_connect`, `disconnect_stripe`

**API keys**
- `api_key_self_revoke` — agent destroys its own credential (no multi-key CRUD in MCP)

## Readiness-aware responses

Every account, listing, and order response includes a `readiness` object with an `actions` array. Each action has a `code`, a `kind` (`api` or `human`), a human-readable `message`, and a `resolve` block with the method and endpoint to call next.

Agents should read `readiness.next` to know what to do after each step. See [Readiness](/readiness).
