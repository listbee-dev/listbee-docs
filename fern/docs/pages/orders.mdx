An order is created when a buyer starts a Stripe Checkout session. It captures the buyer's email, the listing being purchased, and the amount paid. Orders move through a status lifecycle that ends at `fulfilled`, `canceled`, or `failed`.

## How it works

An order begins as `pending` the moment a buyer submits the checkout form. It transitions to `paid` when Stripe confirms payment. From there:

- If the listing has deliverables, ListBee auto-delivers and transitions the order to `fulfilled`.
- If the listing has no deliverables, the order stays `paid` and ListBee fires `order.paid` to your webhook. Your agent handles delivery â€” or calls `POST /v1/orders/{id}/fulfill` to have ListBee deliver.

Orders have two independent status fields: `status` (fulfillment lifecycle) and `payment_status` (Stripe payment state). An order can be `fulfilled` with `payment_status: paid`, then later have `payment_status: refunded` if a refund is issued.

Each order includes `listing_snapshot` and `seller_snapshot` â€” immutable copies of the listing and seller data captured at payment time. These are the authoritative record of what was purchased, regardless of later changes to the listing.

## Fetching an order

<CodeBlocks>
  <CodeBlock title="Python">
```python
from listbee import ListBee

client = ListBee(api_key="lb_...")

order = client.orders.get(order_id="ord_9xM4kP7nR2qT5wY1")

print(order.status)          # fulfilled
print(order.payment_status)  # paid
print(order.buyer_email)     # buyer@example.com
print(order.amount)          # 1900 ($19.00)
print(order.has_deliverables) # True
```
  </CodeBlock>
  <CodeBlock title="curl">
```bash
curl https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1 \
  -H "Authorization: Bearer lb_..."
```

## Fulfilling an order

Call `fulfill` on any `paid` order to push content to the buyer. ListBee creates an access grant, emails the buyer a download link, and transitions the order to `fulfilled`. Idempotent â€” calling it on an already-`fulfilled` order is a no-op.

<CodeBlocks>
  <CodeBlock title="Python">
```python
# After generating content in response to an order.paid webhook:
client.orders.fulfill(
    order_id="ord_9xM4kP7nR2qT5wY1",
    deliverables=[
        {"type": "text", "value": "Your personalized report: ..."},
    ],
)
```
  </CodeBlock>
  <CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1/fulfill \
  -H "Authorization: Bearer lb_..." \
  -H "Content-Type: application/json" \
  -d '{
    "deliverables": [
      {"type": "text", "value": "Your personalized report: ..."}
    ]
  }'
```
  </CodeBlock>
</CodeBlocks>

Deliverable types are `file` (token from `POST /v1/files`), `url`, or `text`. Up to 3 per order.

## Issuing a refund

<CodeBlocks>
  <CodeBlock title="Python">
```python
order = client.orders.refund(order_id="ord_9xM4kP7nR2qT5wY1")

print(order.payment_status)  # refunded
print(order.refund_amount)   # 1900
```
  </CodeBlock>
  <CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1/refund \
  -H "Authorization: Bearer lb_..."
```
  </CodeBlock>
</CodeBlocks>

Only orders with `status: paid` or `status: fulfilled` can be refunded. Refunds are processed through Stripe on the seller's connected account. The platform fee is not returned to the seller.

## Order status lifecycle

| Status | Meaning |
|--------|---------|
| `pending` | Checkout started; buyer data captured; awaiting payment |
| `paid` | Stripe confirmed payment; delivery in progress |
| `fulfilled` | Content delivered to buyer |
| `canceled` | Checkout abandoned or payment failed after 24h |
| `failed` | System error during payment or delivery |

## Actions

`paid` orders include a `readiness.actions` array listing suggested next steps. When no actions apply, the field is omitted.

A `FULFILL_ORDER` action appears on `paid` orders that have no deliverables â€” it signals that your agent can push content via `POST /v1/orders/{id}/fulfill`.

```json
{
  "object": "order",
  "id": "ord_9xM4kP7nR2qT5wY1",
  "status": "paid",
  "has_deliverables": false,
  "readiness": {
    "actions": [
      {
        "code": "FULFILL_ORDER",
        "kind": "api",
        "priority": "suggested",
        "message": "Push deliverables to fulfill this order.",
        "resolve": {
          "method": "POST",
          "endpoint": "/v1/orders/ord_9xM4kP7nR2qT5wY1/fulfill"
        }
      }
    ],
    "next": "FULFILL_ORDER"
  }
}
```

## Key fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique order ID (`ord_` prefixed) |
| `listing_id` | string | ID of the purchased listing |
| `buyer_email` | string | Buyer's email address from Stripe Checkout |
| `amount` | integer | Amount paid in cents |
| `currency` | string | Three-letter ISO currency code, uppercase |
| `status` | string | Fulfillment lifecycle status |
| `payment_status` | string | Stripe payment state: `unpaid`, `paid`, or `refunded` |
| `has_deliverables` | boolean | Whether the listing had deliverables at time of purchase |
| `checkout_data` | object | Custom field values submitted by the buyer |
| `listing_snapshot` | object | Immutable copy of listing data at payment time |
| `seller_snapshot` | object | Immutable copy of seller identity at payment time |
| `deliverables` | array | Content attached to this order |
| `actions` | array | Suggested next steps for `paid` orders; omitted when empty |
| `stripe_payment_intent_id` | string | Stripe PaymentIntent ID for cross-referencing |
| `platform_fee` | integer | Platform fee deducted in cents |
| `refund_amount` | integer | Amount refunded in cents; `0` if not refunded |
| `paid_at` | string | When payment was confirmed |
| `fulfilled_at` | string | When order was fulfilled |
