An order is created when Stripe confirms payment. It captures the buyer's email, the listing purchased, the amount paid, and an immutable snapshot of the listing and seller at that moment.

## Lifecycle

```
PAID  →  FULFILLED
```

- **`paid`** — Stripe confirmed payment. Order exists. For `static` listings, the next step is automatic. For `async` listings, the agent calls `/fulfill`.
- **`fulfilled`** — content has been delivered. `deliverable` and `fulfilled_at` are set.

No `pending`, no `canceled`, no `failed`. Pre-payment state lives in Stripe's Checkout Session — no ListBee order exists until the `checkout.session.completed` webhook fires.

Two orthogonal axes run in parallel to this lifecycle:

- **`payment_status`** — `paid` or `refunded`. A `fulfilled` order can still be refunded.
- **`dispute_status`** — `null`, `needs_response`, `under_review`, `won`, `lost`. Managed by Stripe.

A healthy, reconciled order satisfies all three:

```
status == fulfilled  AND  payment_status == paid  AND  dispute_status IS NULL
```

## Fetching an order

<CodeBlocks>
<CodeBlock title="Python">
```python
order = client.orders.get(order_id="ord_9xM4kP7nR2qT5wY1")
print(order.status)          # paid | fulfilled
print(order.payment_status)  # paid | refunded
print(order.amount)          # 1900 ($19.00)
print(order.unlock_url)      # permanent bearer link
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1 \
  -H "Authorization: Bearer lb_..."
```
</CodeBlock>
</CodeBlocks>

## Listing orders

Cursor-paginated. Filter by `status` (`paid` | `fulfilled`) or `listing_id`.

<CodeBlocks>
<CodeBlock title="Python">
```python
result = client.orders.list(status="paid")

for order in result.data:
    print(order.id, order.buyer_email, order.amount)

if result.has_more:
    next_page = client.orders.list(status="paid", cursor=result.cursor)
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl "https://api.listbee.so/v1/orders?status=paid" \
  -H "Authorization: Bearer lb_..."
```
</CodeBlock>
</CodeBlocks>

## Fulfilling

`POST /v1/orders/{id}/fulfill` pushes the generated content to a `paid` order. ListBee stores the deliverable on the order, emails the buyer the `unlock_url`, and transitions the order to `fulfilled`.

<CodeBlocks>
<CodeBlock title="Python">
```python
from listbee import Deliverable

client.orders.fulfill(
    order_id="ord_9xM4kP7nR2qT5wY1",
    deliverable=Deliverable.text("Your personalized report: ..."),
    metadata={"generated_by": "agent-v3", "model": "claude-opus-4.7"},
)
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1/fulfill \
  -H "Authorization: Bearer lb_..." \
  -H "Content-Type: application/json" \
  -d '{
    "deliverable": {"type": "text", "value": "Your personalized report: ..."},
    "metadata": {"generated_by": "agent-v3"}
  }'
```
</CodeBlock>
</CodeBlocks>

Idempotent: calling `/fulfill` again with the same deliverable on an already-`fulfilled` order is a no-op. Only one deliverable per order — see [Deliverables](/deliverables).

## Unlock URL

Every `paid` order has an `unlock_url` that takes the buyer to their content:

- `url` deliverables → 302 redirect to the stored URL
- `text` deliverables → rendered Markdown page
- `paid` but not yet `fulfilled` (async) → processing page that polls until content arrives

The URL is a permanent bearer link. No expiry, no view counter, no per-buyer revocation. Anyone with the link can access the content. If you need stricter access control, host the content at a URL that enforces it.

## Redelivering webhooks

If your `agent_callback_url` was down or misconfigured when an order happened, requeue the delivery:

<CodeBlocks>
<CodeBlock title="Python">
```python
ack = client.orders.redeliver(order_id="ord_9xM4kP7nR2qT5wY1")
print(ack.scheduled_attempts)  # count of events re-queued
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1/redeliver \
  -H "Authorization: Bearer lb_..." \
  -H "Idempotency-Key: retry-2026-04-18-ord_9xM4kP7nR2qT5wY1"
```
</CodeBlock>
</CodeBlocks>

Response:

```json
{"object": "redelivery_ack", "order_id": "ord_9xM4kP7nR2qT5wY1", "scheduled_attempts": 2}
```

Rate limits: 10 redeliveries/hour per order, 100/hour per API key. If the listing has no `agent_callback_url`, `scheduled_attempts` is `0` — use `GET /v1/events` to poll instead. See [Events](/events).

## Refunding

Full refund only. The platform fee is not returned to the seller.

<CodeBlocks>
<CodeBlock title="Python">
```python
order = client.orders.refund(order_id="ord_9xM4kP7nR2qT5wY1")
print(order.payment_status)  # refunded
print(order.refund_amount)   # 1900
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl -X POST https://api.listbee.so/v1/orders/ord_9xM4kP7nR2qT5wY1/refund \
  -H "Authorization: Bearer lb_..."
```
</CodeBlock>
</CodeBlocks>

Refunding emits `order.refunded`. The order's `status` (paid/fulfilled) does not change — only `payment_status` does.

## Metadata

`metadata` on an order is a free-form dict:

- Listing metadata is snapshotted onto the order at checkout time.
- `/fulfill` can attach additional metadata (e.g. generation parameters).

Stripe-aligned limits: max 50 keys, keys ≤ 40 chars, string values ≤ 500 chars, no nesting. Flows through to webhook payloads.

## Key fields

| Field                   | Type    | Description |
|-------------------------|---------|-------------|
| `id`                    | string  | Unique order ID (`ord_`) |
| `listing_id`            | string  | Purchased listing |
| `status`                | string  | `paid` or `fulfilled` |
| `payment_status`        | string  | `paid` or `refunded` |
| `dispute_status`        | string  | `null`, `needs_response`, `under_review`, `won`, `lost` |
| `deliverable`           | object  | Content delivered (null until `fulfilled`) |
| `unlock_url`            | string  | Permanent bearer link to the unlock page |
| `buyer_email`           | string  | From Stripe Checkout |
| `amount`                | integer | Cents |
| `checkout_data`         | object  | Buyer's submitted custom fields |
| `metadata`              | object  | Forwarded in webhook payloads |
| `listing_snapshot`      | object  | Immutable listing data at payment time |
| `seller_snapshot`       | object  | Immutable seller identity at payment time |
| `readiness`             | object  | `fulfilled` flag + `actions` |
| `stripe_payment_intent_id` | string | For cross-referencing in Stripe |
| `refund_amount`         | integer | `0` if not refunded |
