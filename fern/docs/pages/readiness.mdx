Every resource that has prerequisites — accounts, listings, orders — includes a `readiness` object in its API response. Readiness tells an agent two things: whether the resource is ready, and if not, the concrete steps to get there.

Agents consume readiness programmatically so business rules never need to be hardcoded. When the rules change, the `actions` array updates automatically.

## The readiness object

| Entity  | Flag field     | `true` means                                |
|---------|---------------|---------------------------------------------|
| Account | `operational` | Can accept payments                         |
| Listing | `sellable`    | Buyers can complete a purchase              |
| Order   | `fulfilled`   | Buyer has received the content              |

When the flag is `true`, `actions` is empty and `next` is `null`. When `false`, `actions` lists the steps to resolve.

```json
{
  "readiness": {
    "sellable": false,
    "actions": [
      {
        "code": "stripe_connect_required",
        "kind": "human",
        "priority": "required",
        "message": "Complete Stripe Connect onboarding to accept payments.",
        "resolve": {"method": "redirect", "url": "https://connect.stripe.com/setup/s/..."}
      }
    ],
    "next": "stripe_connect_required"
  }
}
```

## Action shape

| Field              | Type     | Description |
|--------------------|----------|-------------|
| `code`             | string   | Stable machine identifier — branch on this |
| `kind`             | string   | `api` (agent calls directly) or `human` (needs a browser) |
| `priority`         | string   | `required` (blocks progress) or `suggested` (optional) |
| `message`          | string   | Human-readable explanation, safe to relay |
| `resolve.method`   | string   | HTTP method, or `redirect` for human actions |
| `resolve.endpoint` | string   | API path for `kind: api` |
| `resolve.url`      | string   | Full URL for `kind: human` |
| `resolve.params`   | object   | Example request body for API actions |

`next` points to the highest-priority code. When an `api` action exists, it's preferred over a `human` action — the agent can make progress without waiting.

## Action code catalog

The full set — eight codes, grouped by entity:

### Account

| Code                         | Kind   | Resolve |
|------------------------------|--------|---------|
| `otp_verification_pending`   | human  | Human pastes the OTP back to the agent — see [Authentication](/authentication) |
| `stripe_connect_required`    | human  | Redirect the human to `stripe_onboarding_url` |
| `stripe_charges_disabled`    | human  | Human logs into the Stripe dashboard to resolve the flag |
| `account_deleted`            | —      | Terminal — account soft-deleted, no action possible |

### Listing

| Code                          | Kind   | Resolve |
|-------------------------------|--------|---------|
| `listing_unpublished`         | api    | `POST /v1/listings/{id}/publish` |
| `listing_deliverable_missing` | api    | `PATCH /v1/listings/{id}` with a `deliverable` (static listings only) |

### Order

| Code                  | Kind   | Resolve |
|-----------------------|--------|---------|
| `fulfillment_pending` | api    | `POST /v1/orders/{id}/fulfill` — async listings awaiting content |
| `dispute_open`        | human  | Human responds to the dispute in the Stripe dashboard |

## Account readiness

`GET /v1/account` → `readiness.operational`.

<CodeBlocks>
<CodeBlock title="Python">
```python
account = client.account.get()
if account.readiness.operational:
    print("Account is ready to sell")
else:
    for action in account.readiness.actions:
        print(f"[{action.priority}] {action.code}: {action.message}")
```
</CodeBlock>
<CodeBlock title="curl">
```bash
curl https://api.listbee.so/v1/account \
  -H "Authorization: Bearer lb_..."
```
</CodeBlock>

## Listing readiness

`GET /v1/listings/{id}` → `readiness.sellable`.

A listing is sellable when the owning account is operational, the listing is `published`, and the fulfillment path is complete (static mode has a `deliverable`; async mode always qualifies).

```python
listing = client.listings.get(listing_id="lst_7kQ2xY9mN3pR5tW1vB8a")

if not listing.readiness.sellable:
    for action in listing.readiness.actions:
        print(f"[{action.priority}] {action.code}: {action.message}")
```

`listing_unpublished` is emitted last — only after all other blockers are cleared. An agent can iterate `actions` in order and the list converges to `sellable: true`.

## Order readiness

`GET /v1/orders/{id}` → `readiness.fulfilled`.

An `async` order in `paid` status emits `fulfillment_pending` (priority `required`) until the agent calls `/fulfill`. A `static` order is always fulfilled automatically, so this action never appears.

```json
{
  "object": "order",
  "id": "ord_9xM4kP7nR2qT5wY1",
  "status": "paid",
  "readiness": {
    "fulfilled": false,
    "actions": [
      {
        "code": "fulfillment_pending",
        "kind": "api",
        "priority": "required",
        "message": "Deliver content to the buyer via POST /v1/orders/{id}/fulfill.",
        "resolve": {
          "method": "POST",
          "endpoint": "/v1/orders/ord_9xM4kP7nR2qT5wY1/fulfill"
        }
      }
    ],
    "next": "fulfillment_pending"
  }
}
```

When a dispute opens, `dispute_open` appears (priority `required`, kind `human`) — the seller must act in the Stripe dashboard.

## Driving setup with an agent

Because `kind` distinguishes agent-executable steps from human-gated ones, an agent can automate setup end-to-end, stopping only when a human is required:

<CodeBlocks>
<CodeBlock title="Python">
```python
account = client.account.get()
while not account.readiness.operational:
    action = next((a for a in account.readiness.actions if a.kind == "api"), None)
    if action is None:
        # Remaining actions are human-only — hand off
        human = account.readiness.actions[0]
        print(f"Human needed: {human.message}")
        print(f"URL: {human.resolve.url}")
        break
    # Execute the api action, then re-fetch
    execute(action)
    account = client.account.get()
```
</CodeBlock>
</CodeBlocks>

The loop is always: fetch → check flag → execute next `api` action → re-fetch.

## Readiness and errors

When an operation fails because of unmet readiness, the error response carries the same `actions` array. The agent doesn't need a follow-up GET to discover what went wrong.

```json
{
  "type": "https://docs.listbee.so/errors/publish-failed",
  "title": "Conflict",
  "status": 409,
  "detail": "Listing cannot be published: missing deliverable",
  "code": "publish_failed",
  "actions": [
    {
      "code": "listing_deliverable_missing",
      "kind": "api",
      "priority": "required",
      "message": "Static listings require a deliverable before publishing.",
      "resolve": {
        "method": "PATCH",
        "endpoint": "/v1/listings/lst_.../",
        "params": {"deliverable": {"type": "url", "value": "https://..."}}
      }
    }
  ],
  "next": "listing_deliverable_missing"
}
```
