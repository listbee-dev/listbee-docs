The `readiness` object appears on both account and listing responses. It tells you whether the resource is operational and, if not, exactly what to do next. Agents can consume readiness programmatically to drive setup workflows without hardcoding business rules.

## How it works

Every account, listing, and order response includes a `readiness` object. It has three fields:

- **`operational`** (account) or **`sellable`** (listing) — `true` when the resource is ready to use.
- **`actions`** — array of steps to take, each with a code, kind, message, and resolve instructions.
- **`next`** — the code of the highest-priority action. `null` when ready. API-kind actions take priority over human-kind.

When `actions` is empty, the resource is ready. When `actions` is non-empty, work through them in order — each resolved action may unlock the next.

## Action shape

```json
{
  "code": "connect_stripe",
  "kind": "api",
  "message": "Connect Stripe account to accept payments.",
  "resolve": {
    "method": "POST",
    "endpoint": "/v1/account/stripe/connect",
    "url": null,
    "params": null
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `code` | string | Machine-readable identifier for branching logic |
| `kind` | string | `api` — agent can call the endpoint directly; `human` — requires browser redirect |
| `priority` | string | `required` — blocks progress; `suggested` — available opportunity |
| `message` | string | Human-readable explanation; safe to relay to users |
| `resolve.method` | string | HTTP method, or `redirect` for human actions |
| `resolve.endpoint` | string | API path for `kind: api` actions |
| `resolve.url` | string | Full URL for `kind: human` actions |
| `resolve.params` | object | Example request body for API actions |

## Account readiness

`GET /v1/account` returns `readiness.operational`. An account is operational when Stripe is connected, charges are enabled, and billing is current.

<CodeBlocks>
  <CodeBlock title="Python">
```python
from listbee import ListBee

client = ListBee(api_key="lb_...")

account = client.account.get()

if not account.readiness.operational:
    next_action = account.readiness.next
    actions = {a.code: a for a in account.readiness.actions}

    action = actions[next_action]
    print(f"Next step: {action.message}")
    # next_action == "connect_stripe" → call POST /v1/account/stripe/connect
    # next_action == "enable_charges" → redirect human to Stripe onboarding
    # next_action == "update_billing" → redirect human to billing page
```
  </CodeBlock>
  <CodeBlock title="curl">
```bash
curl https://api.listbee.so/v1/account \
  -H "Authorization: Bearer lb_..."
```
  </CodeBlock>
</CodeBlocks>

## Listing readiness

`GET /v1/listings/{id}` returns `readiness.sellable`. A listing is sellable when the account is operational and all listing-specific requirements are met.

<CodeBlocks>
  <CodeBlock title="Python">
```python
listing = client.listings.get(listing_id="lst_7kQ2xY9mN3pR5tW1vB8a")

if not listing.readiness.sellable:
    for action in listing.readiness.actions:
        print(f"{action.code} ({action.kind}): {action.message}")
        if action.kind == "api":
            print(f"  → {action.resolve.method} {action.resolve.endpoint}")
        else:
            print(f"  → redirect to {action.resolve.url}")
```
  </CodeBlock>
  <CodeBlock title="curl">
```bash
curl https://api.listbee.so/v1/listings/lst_7kQ2xY9mN3pR5tW1vB8a \
  -H "Authorization: Bearer lb_..."
```

## Order readiness

`GET /v1/orders/{id}` and order list responses include `readiness.fulfilled`. An order is fulfilled when content has been delivered to the buyer.

`paid` orders that are not yet fulfilled include a `fulfill_order` action — your agent can push content via `POST /v1/orders/{id}/fulfill`.

```json
{
  "object": "order",
  "id": "ord_9xM4kP7nR2qT5wY1",
  "status": "paid",
  "readiness": {
    "fulfilled": false,
    "actions": [
      {
        "code": "fulfill_order",
        "kind": "api",
        "priority": "suggested",
        "message": "Deliver content to buyer via ListBee",
        "resolve": {
          "method": "POST",
          "endpoint": "/v1/orders/ord_9xM4kP7nR2qT5wY1/fulfill"
        }
      }
    ],
    "next": null
  }
}
```

`fulfill_order` uses `priority: suggested` rather than `required` — delivery via ListBee is optional. Sellers can handle delivery externally.

## Common action codes

| Code | Kind | When it appears | Resolution |
|------|------|----------------|------------|
| `connect_stripe` | `api` | No Stripe account connected | `POST /v1/account/stripe/connect` — returns onboarding URL |
| `enable_charges` | `human` | Stripe connected but onboarding incomplete | Redirect to Stripe onboarding via console |
| `update_billing` | `human` | Billing past due or unpaid | Redirect to billing page in console |
| `configure_fulfillment` | `api` | External listing has no fulfillment path (no deliverables, no webhook, no fulfillment_url) | `PATCH /v1/listings/{id}` with `fulfillment_url` |
| `publish_listing` | `api` | All other blockers cleared; listing still in draft | `POST /v1/listings/{id}/publish` |
| `webhook_disabled` | `api` | Webhook was auto-disabled after consecutive failures | Re-enable or replace the webhook |
| `fulfill_order` | `api` | PAID order not yet fulfilled | `POST /v1/orders/{id}/fulfill` |

## Walking an agent through setup

Because `kind` tells you whether a step is automated or human-gated, agents can handle setup end-to-end up to the first human step:

```python
account = client.account.get()

for action in account.readiness.actions:
    if action.kind == "api":
        # Agent executes directly
        print(f"Calling {action.resolve.method} {action.resolve.endpoint}")
    else:
        # Hand off to human
        print(f"Human action required: {action.message}")
        print(f"URL: {action.resolve.url}")
        break
```

`publish_listing` always appears last in the actions list, only when all other blockers are cleared. This pattern — resolve each action and re-fetch — converges to `sellable: true`.

## Readiness and errors

When an operation fails because of unmet readiness (e.g. publishing an incomplete listing), the error response mirrors the readiness structure. The `publish_failed` error includes the same `actions` and `next` fields — your agent doesn't need to make a follow-up GET to discover what's wrong.
